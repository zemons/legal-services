# Case Storage & Document Management — Design Specification

## 1. หลักการ: แยก 2 ที่ชัดเจน

ระบบจัดเก็บเอกสารแบ่งเป็น 2 ส่วนที่ **แยกออกจากกันอย่างสิ้นเชิง**:

| | คลังกลาง (Shared Knowledge) | เอกสารคดี (Case Files) |
|--|---|---|
| **ที่อยู่** | `/opt/legal-services/data/knowledge/` | `/opt/legal-cases/` |
| **จัดการ** | Git repo | Resilio Sync |
| **เข้าถึง** | ทุกคนในองค์กร (read) | เฉพาะทีมคดีนั้น |
| **ข้อมูล** | ฎีกา, กฎหมาย, templates, strategy_patterns | เอกสารลูกค้า, พยาน, ร่างคำฟ้อง |
| **ขนาด** | คงที่ (~50 MB) | โตเรื่อยๆ (ไม่จำกัด) |
| **ความลับ** | ไม่ลับ (ภายในองค์กร) | ลับเฉพาะคดี |
| **AI ใช้ยังไง** | semantic search (RAG) | read ทั้ง folder → วิเคราะห์ |

### Metadata ของ Knowledge

ทุกไฟล์ใน knowledge/ ต้องมี YAML frontmatter:

```markdown
---
type: dika
case_no: 1234/2566
court: ศาลฎีกา
year: 2566
category: แพ่ง
topic: [สัญญาเช่า, ผิดนัด, ค่าเสียหาย]
summary: ผู้เช่าผิดนัดชำระค่าเช่า ผู้ให้เช่าบอกเลิกสัญญาได้
result: โจทก์ชนะ
legal_basis: [ป.พ.พ. ม.560, ป.พ.พ. ม.387]
key_facts: [มีสัญญาลายลักษณ์อักษร, ผู้เช่าไม่ชำระ 3 เดือน, บอกกล่าวแล้ว]
ruling_principle: ผู้ให้เช่ามีสิทธิบอกเลิกสัญญาเมื่อผู้เช่าผิดนัด
---
```

## 3. โครงสร้างเอกสารคดี

### สร้างอัตโนมัติเมื่อรับคดี

เมื่อ Odoo สร้าง CRM Lead + Case ID → ระบบสร้าง folder อัตโนมัติ:

```
/opt/legal-cases/
└── {case_id}/                         ← เช่น 2566-0042
    │
    ├── _meta.json                     ← ข้อมูลคดีจาก Odoo (sync)
    │
    ├── _ai/                           ← AI เขียนอัตโนมัติ
    │   ├── case_summary.md            ← สรุปคดีภาพรวม
    │   ├── strategy.md                ← แนวคดี (IRAC 2-3 มุม)
    │   ├── related_dika.md            ← ฎีกาที่เกี่ยวข้อง (copy สรุปมา)
    │   ├── related_statute.md         ← มาตราที่เกี่ยวข้อง
    │   ├── evidence_checklist.md      ← รายการพยานที่ต้องเตรียม
    │   ├── timeline.md                ← ลำดับเหตุการณ์
    │   └── document_index.json        ← รายการเอกสารทั้งหมด + สรุป
    │
    ├── intake/                        ← เอกสารขาเข้าจากลูกค้า
    │
    ├── evidence/                      ← พยานหลักฐาน
    │   ├── witness/                   ← พยานบุคคล (บันทึกคำให้การ)
    │   ├── document/                  ← พยานเอกสาร

    │
    ├── correspondence/                ← หนังสือโต้ตอบ

    └── court/                         ← เอกสารจากศาล
```

## 4. AI วิเคราะห์เอกสารคดี

### Flow: เอกสารใหม่เข้ามา

```
ทนายลากไฟล์ "สัญญาเช่า.pdf" ใส่ ~/cases/2566-0042/intake/
    │
    ▼
Resilio Sync → server /opt/legal-cases/2566-0042/intake/
    │
    ▼
Resilio API (file event) → Odoo controller
    │
    ▼
Odoo → adkcode: POST /case/analyze
    {
      "case_id": "2566-0042",
      "new_file": "intake/สัญญาเช่า.pdf",
      "event": "file_added"
    }
    │
    ▼
┌─────────────────────────────────────────────┐
│  adkcode case_strategist                     │
│                                              │
│  Step 1: อ่านเอกสารเฉพาะคดี                   │
│    scope: /app/data/cases/2566-0042/          │
│    - อ่าน _meta.json → รู้ประเภทคดี            │
│    - อ่านเอกสารใหม่ (OCR ถ้าเป็นรูป)           │
│    - อ่านเอกสารอื่นที่มีอยู่แล้ว                  │
│                                              │
│  Step 2: ค้นคลังกลาง                          │
│    scope: /app/data/knowledge/                │
│    - legal_search("สัญญาเช่า ผิดนัด")          │
│      → dika/ ได้ฎีกาที่คล้าย 5 อันดับ           │
│    - legal_search("ม.560 เช่าทรัพย์")           │
│      → statute/ ได้มาตราที่เกี่ยวข้อง            │
│    - legal_search_by_facts(                   │
│        ["มีสัญญาลายลักษณ์อักษร",              │
│         "ผิดนัดชำระ 3 เดือน"])                  │
│      → strategy_patterns/ ได้แนวคดีเดิม        │
│                                              │
│  Step 3: วิเคราะห์ IRAC                       │
│    - มุม 1: ผิดสัญญา (ม.560)                  │
│    - มุม 2: ละเมิด (ม.420)                    │
│    - มุม 3: เจรจา/ไกล่เกลี่ย                    │
│    - เปรียบเทียบ → แนะนำมุมที่ดีที่สุด           │
│                                              │
│  Step 4: เขียนผลลง _ai/                      │
│    scope: /app/data/cases/2566-0042/_ai/      │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
_ai/ folder sync กลับเครื่องทนาย + LINE push แจ้ง
```

### ทนายสั่งวิเคราะห์เอง (On-demand)

นอกจาก event-driven ทนายสามารถสั่งวิเคราะห์ได้เอง:

```
วิธี 1: LINE chat
  ทนาย: "วิเคราะห์แนวคดี 2566-0042"
  → bot → adkcode case_strategist
  → ตอบกลับใน LINE + เขียน _ai/strategy.md

วิธี 2: LIFF /liff/cases
  กดปุ่ม [วิเคราะห์แนวคดี] ที่หน้า case detail
  → Odoo controller → adkcode
  → แสดงผลใน LIFF + เขียน _ai/strategy.md

วิธี 3: Odoo backend
  ทนายกดปุ่ม [AI วิเคราะห์] ในหน้า CRM Lead
  → Odoo → adkcode
  → แสดงผลใน Odoo + เขียน _ai/strategy.md
```

### _ai/case_summary.md

```markdown
# สรุปคดี: 2566-0042

**ประเภท:** แพ่ง — สัญญาเช่า
**สถานะ:** กำลังดำเนินการ
**ลูกค้า:** นาย ก. (ผู้ให้เช่า/โจทก์)
**คู่กรณี:** นาย ข. (ผู้เช่า/จำเลย)

## ข้อเท็จจริง

1. นาย ก. ให้นาย ข. เช่าบ้านเลขที่ 123 ค่าเช่าเดือนละ 15,000 บาท
2. มีสัญญาเช่าเป็นลายลักษณ์อักษร ลงวันที่ 1 ม.ค. 2566
3. นาย ข. ชำระค่าเช่าถึงเดือน มิ.ย. 2566 แล้วหยุดชำระ
4. นาย ก. ส่งจดหมายทวงถามลงทะเบียน 2 ครั้ง (ก.ค. และ ส.ค.)
5. นาย ข. ไม่ตอบและยังคงอยู่ในบ้านเช่า

## ค่าเสียหายเบื้องต้น

- ค่าเช่าค้าง: 3 เดือน × 15,000 = 45,000 บาท
- ดอกเบี้ย: ตาม ป.พ.พ. ม.224 (ร้อยละ 5 ต่อปี)

## อายุความ

- สัญญาเช่า: 5 ปี (ป.พ.พ. ม.193/33)
- หมดอายุความ: ประมาณ ก.ค. 2571

## เอกสารที่มี

| # | ไฟล์ | ประเภท | สรุป |
|---|------|--------|------|
| 1 | intake/สัญญาเช่า.pdf | สัญญา | สัญญาเช่า 1 ปี ค่าเช่า 15,000/เดือน |
| 2 | intake/จดหมายทวงถาม.jpg | จดหมาย | ทวงถามครั้งที่ 1 ลงวันที่ 15 ก.ค. 2566 |

*อัปเดตล่าสุด: 2026-03-14 โดย AI*
*⚠️ ข้อมูลนี้เป็นสรุปอัตโนมัติ ต้องให้ทนายตรวจสอบ*
```

### _ai/strategy.md

```markdown
# วิเคราะห์แนวคดี: 2566-0042

## มุมที่ 1: ผิดสัญญาเช่า (ป.พ.พ. ม.560)

**ISSUE:** จำเลยผิดสัญญาเช่าโดยไม่ชำระค่าเช่าหรือไม่?

**RULE:**
- ป.พ.พ. ม.560: ผู้เช่าจำต้องชำระค่าเช่า
- ป.พ.พ. ม.387: ถ้าคู่สัญญาฝ่ายหนึ่งไม่ชำระหนี้ อีกฝ่ายหนึ่งจะกำหนด
  ระยะเวลาพอสมควรแล้วบอกเลิกสัญญาได้
- องค์ประกอบ: (1) มีสัญญาเช่า (2) ถึงกำหนดชำระ (3) ไม่ชำระ (4) บอกกล่าวแล้ว

**APPLICATION:**
- ✅ มีสัญญาเช่าเป็นลายลักษณ์อักษร (พยานเอกสาร: สัญญาเช่า.pdf)
- ✅ ค่าเช่าครบกำหนดทุกวันที่ 1 ของเดือน
- ✅ ไม่ชำระ 3 เดือนติดต่อกัน (ก.ค.–ก.ย. 2566)
- ✅ บอกกล่าวแล้ว 2 ครั้ง (จดหมายลงทะเบียน)
- ฎีกาสนับสนุน:
  - ฎีกาที่ 1234/2566: ข้อเท็จจริงคล้ายกัน ศาลตัดสินให้โจทก์ชนะ
    หลัก: "ผู้ให้เช่ามีสิทธิบอกเลิกสัญญาเมื่อผู้เช่าผิดนัด"

**CONCLUSION:** โอกาสชนะ **สูง**
- จุดแข็ง: พยานหลักฐานครบ มีฎีกาสนับสนุน
- จุดอ่อน: ไม่มี

---

## มุมที่ 2: ละเมิด (ป.พ.พ. ม.420)

**ISSUE:** จำเลยกระทำละเมิดต่อโจทก์หรือไม่?

**RULE:**
- ป.พ.พ. ม.420: ผู้ใดจงใจหรือประมาทเลินเล่อ ทำต่อบุคคลอื่น
  โดยผิดกฎหมาย ให้เขาเสียหาย ต้องใช้ค่าสินไหมทดแทน
- องค์ประกอบ: (1) กระทำ (2) จงใจหรือประมาท (3) ผิดกฎหมาย (4) เสียหาย (5) สัมพันธ์เหตุผล

**APPLICATION:**
- ⚠️ การไม่ชำระค่าเช่าเป็นเรื่อง "ผิดสัญญา" ไม่ใช่ "ผิดกฎหมาย" โดยตรง
- ❌ ยากพิสูจน์ว่าจำเลย "จงใจ" ทำละเมิด
- ฎีกาที่ 5678/2565: "การไม่ชำระหนี้ตามสัญญาเป็นเรื่องผิดสัญญา ไม่ใช่ละเมิด"

**CONCLUSION:** โอกาสชนะ **ต่ำ**
- จุดแข็ง: ไม่มี
- จุดอ่อน: ไม่ตรงองค์ประกอบ ฎีกาขัดแย้ง

---

## มุมที่ 3: เจรจา/ไกล่เกลี่ย

**ข้อดี:**
- ได้เงินเร็วกว่าสู้คดี (ไม่ต้องรอศาล)
- ประหยัดค่าธรรมเนียมศาล + ค่าทนาย
- รักษาความสัมพันธ์

**ข้อเสีย:**
- อาจได้เงินน้อยกว่า (ลดหนี้ให้จำเลย)
- จำเลยอาจไม่ยอมเจรจา

**เหมาะเมื่อ:** จำเลยยังมีทรัพย์สินและยินดีเจรจา
```

## 6. Resilio Sync — การแชร์เอกสารคดี

### สิ่งที่ Sync

```
/opt/legal-cases/                  ← Resilio Sync root
├── 2566-0042/                     ← Shared folder 1 (ทีมคดี 0042)
├── 2566-0099/                     ← Shared folder 2 (ทีมคดี 0099)
└── 2567-0001/                     ← Shared folder 3 (ทีมคดี 0001)
```

**แต่ละ case folder เป็น Resilio shared folder แยก** — ทนายแต่ละคนเข้าถึงเฉพาะคดีที่ตนรับผิดชอบ

### สิ่งที่ไม่ Sync

```
/opt/legal-services/data/knowledge/    ← ไม่ sync (อยู่ server เท่านั้น)
/opt/legal-services/data/templates/    ← ไม่ sync (อยู่ server เท่านั้น)
```

ทนายเข้าถึง knowledge + templates **ผ่าน AI search** ไม่ต้อง sync มาเครื่องตัวเอง

### Access Control

```
Odoo CRM Lead (case 2566-0042)
  └── collaborator_ids: [ทนาย ค., ทนาย ง.]
        │
        ▼
  Resilio API: share folder 2566-0042
  ให้ user: ทนาย ค. (read_write), ทนาย ง. (read_write)
        │
        ▼
  ทนาย ค. เครื่อง PC: ~/cases/2566-0042/  ← sync อัตโนมัติ
  ทนาย ง. เครื่อง PC: ~/cases/2566-0042/  ← sync อัตโนมัติ
  ทนาย จ. (ไม่อยู่ในทีม): ไม่เห็น folder นี้
```


## 7. Docker Volumes

```yaml
services:
  web:
    volumes:
      - ./legal-services/odoo-addons:/mnt/extra-addons
      - ./legal-services/data/templates:/mnt/templates
      - /opt/legal-cases:/mnt/cases                    # case files (แยก)

  adkcode:
    volumes:
      - ./legal-services/adkcode:/app
      - ./legal-services/data/knowledge:/app/data/knowledge:ro   # คลังกลาง (read-only)
      - ./legal-services/data/templates:/app/data/templates:ro   # templates (read-only)
      - /opt/legal-cases:/app/data/cases                         # case files (read-write)
```

### adkcode guardrails

```python
ALLOWED_DIRS = {
    '/app/data/knowledge':  'read',        # คลังกลาง — อ่านอย่างเดียว
    '/app/data/templates':  'read',        # templates — อ่านอย่างเดียว
    '/app/data/cases':      'read_write',  # case files — อ่าน+เขียน (_ai/ เท่านั้น)
}

# AI เขียนได้เฉพาะ _ai/ folder
CASE_WRITE_WHITELIST = ['_ai/']
```

## 8. Feedback Loop — เรียนรู้จากคดีจริง

เมื่อคดีจบ:

```
1. ทนายอัปเดต case_status = "closed_won" ใน Odoo
2. Odoo ถาม: "ต้องการบันทึกแนวคดีเป็น strategy pattern ไหม?"
3. ทนายกดยืนยัน + เพิ่ม notes
4. ระบบสร้าง strategy_pattern:

   /opt/legal-services/data/knowledge/strategy_patterns/

   ---
   type: strategy_pattern
```
